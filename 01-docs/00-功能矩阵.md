# QooERP - 企业级开源ERP系统功能矩阵

QooERP将采用**四层微服务架构**,结合容器化部署和分布式系统技术,实现高可用性和可扩展性。

## 一、系统架构设计

| 层级名称 | 功能 |
|---------|------|
| 表现层 | TypeScript + Vue 3 + TDesign (PC + Mobile), Electron/Capacitor + Vite + Pinia, 100+种语言,主题定制与个性化仪表盘,操作日志回放,批量操作,快捷键配置 |
| 业务逻辑层 | Spring Boot + Nacos + Ribbon + Feign + Sentinel + Nacos + Activiti + 低代码表单设计工具,API文档自动生成器,模块依赖管理器 |
| 数据访问层 | PostgreSQL主数据库,Spring Cache + Caffeine本地缓存,消息队列(数据库轮询),MinIO分布式文件存储,ShardingSphere分片管理,Druid连接池 |
| 基础设施层 | Kubernetes容器编排,HPA自动扩缩容,Istio服务网格,服务健康检查,资源调度优化 |

## 二、DDD领域分层设计

| 层次名称 | 功能模块 |
|---------|---------|
| 用户界面层 | Web前端(Vue 3 + TDesign),移动端(iOS/Android/小程序),API网关,多语言多时区,权限校验与会话管理 |
| 应用层 | 应用服务,命令处理器,查询处理器,事件订阅器,工作流编排,跨领域协调 |
| 领域层 | 聚合根,实体,值对象,领域服务,领域事件,仓储接口,领域规则引擎 |
| 基础设施层 | 仓储实现,数据访问,外部服务集成,消息队列,缓存实现,文件存储 |

## 八、功能规范说明

### 8.1 数据库表命名规则

1. **主表命名规则**
   - 采用`模块_实体名`格式
   - 示例:`hr_employee`(员工表)、`fin_account`(会计科目表)、`scm_supplier`(供应商表)
   - 全部使用小写字母,单词间用下划线分隔

2. **配置表命名规则**
   - 采用`模块_实体名_config`格式
   - 示例:`sys_param_config`(系统参数配置)、`hr_kpi_config`(KPI配置)
   - 用于存储规则、策略、模板等配置信息

3. **记录表命名规则**
   - 采用`模块_实体名_log`或`模块_实体名_record`格式
   - 示例:`sys_login_log`(登录日志)、`hr_attendance_record`(考勤记录)
   - 用于存储历史记录、操作日志、审计信息

4. **关联表命名规则**
   - 采用`模块_关联名称_rel`格式
   - 示例:`hr_role_permission_rel`(角色权限关联)、`scm_order_item_rel`(订单明细关联)
   - 用于维护多对多关系,只存储外键ID

5. **字段命名规则**
   - 主键统一使用`id`
   - 外键采用`关联表_id`格式,如`user_id`、`dept_id`
   - 状态字段使用`status`,统一枚举值
   - 创建时间`create_time`,更新时间`update_time`,逻辑删除`del_flag`
   - 创建人`create_by`,更新人`update_by`

### 8.2 数据库公共字段规范

所有数据库表必须包含以下公共字段:

| 字段名 | 数据类型 | 是否必填 | 默认值 | 说明 |
|--------|----------|----------|--------|------|
| id | BIGSERIAL | 是 | 自动生成 | 主键,BIGSERIAL类型自动递增,分布式场景使用雪花算法 |
| create_time | TIMESTAMP | 是 | CURRENT_TIMESTAMP | 创建时间,记录数据创建时的时间戳,精确到微秒 |
| update_time | TIMESTAMP | 是 | CURRENT_TIMESTAMP | 更新时间,每次数据更新时自动更新,通过触发器实现 |
| create_by | BIGINT | 否 | NULL | 创建人 ID,记录创建该数据的用户ID |
| update_by | BIGINT | 否 | NULL | 更新人 ID,记录最后更新该数据的用户ID |
| del_flag | SMALLINT | 是 | 0 | 逻辑删除标识,0表示未删除,1表示已删除 |
| tenant_id | BIGINT | 否 | NULL | 租户 ID,多租户场景下标识数据所属租户 |
| biz_date | INTEGER | 否 | NULL | 分区字段,格式为 yyyymmdd,用于数据分区和归档 |

**字段说明:**
- `id`: 主键字段,使用PostgreSQL的BIGSERIAL类型自动递增,分布式部署使用雪花算法(Snowflake)生成分布式唯一ID
- `create_time`: 数据创建时间,TIMESTAMP类型,精确到微秒,用于数据追溯和审计
- `update_time`: 数据最后更新时间,TIMESTAMP类型,通过PostgreSQL触发器(TRIGGER)机制自动更新
- `create_by`: 创建操作的用户ID,用于数据溯源和权限控制
- `update_by`: 最后更新操作的用户ID,用于数据追溯
- `del_flag`: 逻辑删除标识,使用SMALLINT类型,不物理删除数据,通过标记字段实现软删除
- `tenant_id`: 多租户隔离字段,在SaaS场景下实现租户数据隔离
- `biz_date`: 业务日期字段,INTEGER类型,用于数据分区、归档、统计分析,格式示例:20260218

**PostgreSQL特殊特性说明:**
1. BIGSERIAL 类型等价于 BIGINT + SEQUENCE,自动生成递增主键
2. TIMESTAMP 类型支持时区,精确到微秒级别
3. 逻辑删除时设置 `del_flag = 1`,不执行 `DELETE` 语句
4. 查询时默认过滤 `del_flag = 0` 的数据
5. `tenant_id` 为 NULL 的数据表示公共数据,所有租户可见
6. `biz_date` 字段用于PostgreSQL分区表(Partitioning)的分区键,提升查询性能

**注意事项:**
1. 所有表(包括主表、配置表、记录表、关联表)都必须包含以上公共字段
2. 公共字段在数据库层面创建后,不在业务代码中重复定义,通过基类实体继承
3. 需要创建触发器实现 `update_time` 字段的自动更新
4. 为 `del_flag`、`tenant_id`、`biz_date` 创建索引以优化查询性能
5. 大数据量表建议使用PostgreSQL原生分区表功能,按 `biz_date` 分区

### 8.2 前端页面规范

1. **页面类型**
   - **列表页**: 展示数据列表,支持查询、分页、排序、批量操作
   - **详情页**: 展示单条数据的完整信息,支持编辑
   - **新增页**: 创建新数据的表单页面
   - **编辑页**: 修改已有数据的表单页面
   - **关联页**: 管理关联关系的页面(如角色权限配置)

2. **页面命名规则**
   - 列表页:`模块-实体名-列表`(如`hr-employee-list`)
   - 详情页:`模块-实体名-详情`(如`hr-employee-detail`)
   - 新增页:`模块-实体名-新增`(如`hr-employee-add`)
   - 编辑页:`模块-实体名-编辑`(如`hr-employee-edit`)
   - 关联页:`模块-关联名称-管理`(如`hr-role-permission-manage`)

3. **必选功能**
   - 所有管理页面必须包含:查询、新增、编辑、删除、导出、导入
   - 配置页面必须包含:新增、编辑、删除、启用/禁用、复制
   - 记录页面必须包含:查询、查看详情、导出、统计图表
   - 关联页面必须包含:添加关联、删除关联、批量操作

4. **页面布局规范**
   - 统一使用TDesign组件库
   - 顶部:页面标题、操作按钮栏
   - 中部:搜索区域(默认可折叠)、数据表格或表单
   - 底部:分页器(列表页)、操作按钮(表单页)
   - 右侧:常用快捷操作面板

5. **数据展示规范**
   - 列表页默认展示10-20个关键字段
   - 详情页展示所有字段,按业务逻辑分组
   - 关联数据通过标签页(Tab)或弹窗展示
   - 状态字段使用标签(Tag)组件展示
   - 金额、数量等数值字段右对齐

### 8.3 功能关系管理规则

1. **关联表设计原则**
   - 仅存储两个外键ID,不存储其他业务字段
   - 联合主键或唯一索引确保不重复关联
   - 必须包含`create_time`和`create_by`字段
   - 逻辑删除,物理关联关系不删除

2. **关联管理页面规则**
   - 左侧列表显示已关联数据,支持搜索和筛选
   - 右侧显示可关联数据,支持批量选择
   - 中间提供添加/移除操作按钮
   - 支持拖拽排序(有顺序要求的关联)
   - 显示关联关系的附加属性(如权限类型、角色类型)

3. **级联操作规则**
   - 主表删除时,必须先删除关联关系
   - 支持级联查询,关联数据通过JOIN一次获取
   - 关联表不级联删除,由业务逻辑控制
   - 重要关联关系删除需要二次确认

4. **数据一致性保证**
   - 使用数据库事务保证关联操作的原子性
   - 关联表增加唯一索引防止重复关联
   - 通过触发器或应用层逻辑保证引用完整性
   - 定期清理已逻辑删除主表的脏关联数据
